# MultiMind Browser
## 多 AI 并行分屏浏览器 — 产品设计文档 v0.1
**2026年6月 · 机密 · 内部使用**

---

## 1. 产品概述

### 1.1 产品定位

MultiMind Browser 是一款基于 Electron（内嵌 Chromium）的桌面浏览器，核心能力是将浏览器窗口分割为多个独立的 WebView 格子，让用户同时打开多个 AI 网页（Claude、ChatGPT、DeepSeek 等），并通过底部统一输入框一次输入、同步发送到所有窗口。

本工具同时也是一个完整可用的浏览器，用户可以像使用 Chrome/Safari 一样用它访问任意网站。

### 1.2 产品背景与真实痛点

Claude Code 与 Codex 桌面版均仅支持 Apple Silicon（M 系列芯片），2020 年前购买的所有 Intel 芯片 Mac 用户均无法使用其桌面应用。同时，这两款工具均为单一模型绑定：Claude Code 只用 Claude，Codex 只用 GPT。它们的记忆是项目级的，不是个人级的。

MultiMind Browser 从三个维度切入：支持 Intel Mac；同时展示多个 AI 的视角；属于用户自己的浏览器工具，而不是某家 AI 厂商的锁定商品。

### 1.3 目标用户

- 需要同时参考多个 AI 意见来深入思考问题的知识工作者
- 因硬件限制（Intel Mac）无法使用 Claude Code / Codex 桌面版的用户
- 希望在多个 AI 之间进行横向对比而无需反复切换标签页的用户
- Windows 用户（同样没有官方多 AI 并行工具）

### 1.4 核心价值主张

> 同一个问题，一次输入，同时听到 Claude、ChatGPT、DeepSeek 等多种声音，无需在标签页之间反复切换。完全使用用户自己已有的 AI 账号，无需额外订阅任何服务。

## 2. 功能范围（MVP）

### 2.1 分屏布局

用户可在工具栏一键切换五种布局模式：

| 布局模式 | 格子数量 | 典型使用场景 | 快捷键 |
|---------|---------|------------|--------|
| 单屏 | 1 | 普通浏览 | Cmd/Ctrl + 1 |
| 左右分屏 | 2 | 两个 AI 对比 | Cmd/Ctrl + 2 |
| 上下分屏 | 2 | 竖屏对比阅读 | Cmd/Ctrl + 3 |
| 三分屏 | 3 | 三个 AI 讨论 | Cmd/Ctrl + 3（切换） |
| 四分屏 | 4 | 四个 AI 同时讨论 | Cmd/Ctrl + 4 |

每个格子是一个独立的 WebView，拥有独立的 Cookie、Session、浏览历史，互不干扰。用户在每个格子里登录各自的 AI 账号，登录状态持久保存。

格子之间用 4px 间隔线分隔，可拖拽调整各格子宽高比例。

### 2.2 统一输入与同步发送

**输入区位置**

浏览器底部常驻一个统一输入框（高度约 80px），在分屏模式下始终可见。单屏普通浏览时可折叠隐藏。

**同步发送流程**

1. 用户在底部输入框输入问题，按 Enter 或点击发送按钮
2. 主进程同时向所有激活格子注入 JavaScript，将文字填入该 AI 网页的输入框
3. 注入完成后自动模拟点击发送按钮（或触发 Enter 键事件）
4. 各格子的 AI 网页独立响应，流式输出各自的回答

> ⚠️ 注意：自动发送依赖对各 AI 网站 DOM 结构的识别，当 AI 官网更新时可能短暂失效。策略是优先尝试自动提交；若检测失败，自动回退为「文字已填入，请手动按 Enter」的提示，确保用户始终能发出消息。

**各 AI 网站注入适配**

| AI 网站 | 输入框识别策略 | 发送触发方式 | 备注 |
|--------|-------------|------------|------|
| Claude (claude.ai) | contenteditable div | 模拟 Enter 键 | 需检测是否有附件上传状态 |
| ChatGPT (chatgpt.com) | textarea / contenteditable | 点击发送按钮 | 需区分 GPT-4o 与旧版 UI |
| DeepSeek (chat.deepseek.com) | textarea | 模拟 Enter 键 | — |

### 2.3 用户自定义格子配置

每个格子加载哪个网站完全由用户决定，应用内置常见 AI 网站清单供快速选择，也支持用户手动输入任意 URL。

**内置 AI 网站清单**

国际：Claude（claude.ai）· ChatGPT（chatgpt.com）· Grok（grok.com）· Perplexity（perplexity.ai）· Copilot（copilot.microsoft.com）

> ⚠️ Gemini 已从内置清单中移除。Gemini 仅支持 Google 账号登录，而 Google 全面禁止嵌入式浏览器（包括 Electron）登录 Google 账号，此限制无法绕过，因此 Gemini 在 MultiMind Browser 中无法正常登录使用。用户仍可通过「自定义 URL」手动添加 gemini.google.com，但添加时会收到风险提示（见 2.6 节）。

国内：DeepSeek（chat.deepseek.com）· Kimi（kimi.moonshot.cn）· 文心一言（yiyan.baidu.com）· 通义千问（tongyi.aliyun.com）· 豆包（www.doubao.com）· 智谱清言（chatglm.cn）

**内置布局模板**（首次使用时引导选择，后续可在设置中修改）

| 模板名 | 格子配置 |
|-------|---------|
| 中美三强 | Claude · ChatGPT · DeepSeek |
| 国产双雄 | DeepSeek · Kimi |
| 美国双雄 | Claude · ChatGPT |
| 自定义 | 用户逐格填写 URL |

用户可为每个格子单独设置默认网址，布局和网址配置本地持久化，下次启动时自动恢复。

**格子不限于 AI 聊天网站，也支持搜索引擎**

每个格子有两种行为模式，用户选择网站时由系统自动判定：

- **Chat 模式**：AI 聊天网站，统一发送时填入对话框并触发提交，原地流式响应
- **Search 模式**：传统搜索引擎，统一发送时直接拼接查询 URL 并导航，整页跳转到搜索结果

内置搜索引擎清单：Google（google.com）· Bing（bing.com）· 百度（baidu.com）· 搜狗（sogou.com）· DuckDuckGo（duckduckgo.com）

用户可以自由混搭，例如左侧两格放 Claude + ChatGPT，右侧两格放 Google + 百度，一次提问后，AI 给出回答的同时，搜索引擎给出对应的搜索结果页，便于交叉验证。

### 2.4 作为通用浏览器使用

MultiMind Browser 是一个完整的浏览器，具备以下标准功能：

- 标准地址栏，支持 URL 输入和搜索（默认搜索引擎可配置）
- 前进 / 后退 / 刷新 / 首页按钮
- 多标签页（Tab）管理
- 书签功能
- 下载管理
- 开发者工具（F12）
- 隐私模式（不保存浏览记录和 Cookie）

### 2.5 格子内操作

每个格子右上角悬浮快捷操作菜单（hover 时显示）：

- 刷新本格子
- 修改本格子地址
- 在新标签页打开当前网页
- 将该格子从同步发送中移除（不发送给该 AI）
- 静音 / 取消静音

### 2.6 用户提示规范

应用内所有提示条统一遵循以下规则：不使用"错误""失败"等引发焦虑的词，用中性语气说明情况并给出下一步建议；提示条可手动关闭，不自动消失；同类提示在同一会话中只显示一次，避免反复打扰。

| 触发场景 | 提示文案方向 | 提示位置 |
|---------|------------|---------|
| 用户在「编辑格子」中手动输入 gemini.google.com 或其他已知仅支持 Google 登录的网址 | 说明该网站仅支持 Google 账号登录，Google 已限制嵌入式浏览器登录，可能无法正常使用 | 输入框下方，确认添加前 |
| 格子内检测到 Google 登录拦截页（`signin/rejected`） | 提示该网站的 Google 登录在内嵌浏览器中受限，建议改用邮箱/密码方式登录（如该网站支持） | 格子上方悬浮条 |
| 统一发送时某格子注入失败 | 提示文字已填入对应格子，请手动按 Enter 发送 | 该格子上方悬浮条 |
| 某格子长时间未响应或加载失败 | 提示该网站当前无法访问，可点击重试 | 格子内居中显示 |

## 4. 技术方案

### 4.1 技术选型

| 层次 | 技术选型 / 理由 |
|-----|--------------|
| 应用框架 | Electron 33+  —  内嵌 Chromium，跨平台（Mac x86/ARM、Windows），成熟生态 |
| 前端 UI | React 18 + TypeScript  —  AI 辅助编码友好，组件化易维护 |
| 样式 | Tailwind CSS  —  快速布局，无需手写大量 CSS |
| WebView 技术 | Electron WebContentsView  —  官方推荐替代已废弃的 BrowserView |
| 主进程通信 | Electron IPC（ipcMain / ipcRenderer）  —  渲染进程与主进程双向通信 |
| 网站注入 | webContents.executeJavaScript()  —  向 WebView 注入输入和发送指令 |
| 数据持久化 | electron-store  —  轻量 JSON 存储用户配置、格子布局、书签 |
| 打包工具 | electron-builder  —  一键生成 .dmg（x86 + ARM）和 .exe 安装包 |

### 4.2 进程架构

Electron 应用分为主进程（Main Process）和渲染进程（Renderer Process）：

- **主进程**：管理应用生命周期、创建 BrowserWindow、管理所有 WebContentsView（即每个分屏格子）、处理 IPC 消息
- **渲染进程（UI层）**：React 应用，负责绘制工具栏、地址栏、底部输入框、布局切换按钮等 Chrome UI
- **WebContentsView（每个格子）**：独立的浏览器实例，加载 AI 网页，接收主进程注入的 JS 脚本

> 关键设计：底部统一输入框在渲染进程中，用户按发送后，渲染进程通过 IPC 把文字发给主进程，主进程再依次调用每个 WebContentsView 的 `executeJavaScript()` 完成注入。

### 4.3 网站注入技术细节

**输入填充**

针对不同 AI 网站，注入 JS 的核心逻辑为：找到输入框元素 → 设置内容 → 触发 React/Vue 等框架所需的 input/change 事件 → 触发发送。

由于各 AI 网站的前端框架会拦截直接赋值，需使用 `Object.getOwnPropertyDescriptor` 获取原生 setter 并手动触发，确保框架识别到输入变化。

**发送触发策略（按优先级）**

1. 点击发送按钮（通过 `aria-label` / `data-testid` 定位，优先最稳定）
2. 模拟 `KeyboardEvent` Enter（适用于仅监听键盘的输入框）
3. 失败回退：在格子上方显示提示条，说明文字已填入，请手动按 Enter

**适配器维护策略**

每个 AI 网站的注入逻辑封装为独立的适配器文件（如 `adapters/claude.ts`、`adapters/chatgpt.ts`），当官网 UI 更新导致失效时，只需更新对应适配器，不影响其他功能。适配器通过应用自动更新机制推送，用户无需手动升级。

### 4.4 跨平台打包方案

| 平台 | 架构 | 产物格式 | 打包命令 |
|-----|-----|--------|--------|
| macOS | x86_64 (Intel) | .dmg | `electron-builder --mac --x64` |
| macOS | arm64 (M1/M2/M3) | .dmg | `electron-builder --mac --arm64` |
| macOS | Universal | .dmg（两架构合并） | `electron-builder --mac --universal` |
| Windows | x64 | .exe（NSIS 安装包） | `electron-builder --win --x64` |

推荐发布 macOS Universal 版本（一个 .dmg 同时支持 Intel 和 Apple Silicon），Windows 发布 x64 .exe。之所以能天然支持 Intel Mac，是因为 Electron 自带 Chromium，不依赖系统安装的任何运行时。

### 4.5 隐私与安全

- 所有用户数据（配置、书签、格子布局）存储在本地，不上传任何服务器
- 各格子 WebView 的 Cookie 和 Session 由 Electron 管理，存储在本地用户目录
- 注入 JS 脚本仅操作输入框和发送按钮，不读取 AI 的回答内容，不做任何数据采集
- 应用不要求任何账号注册，用户直接在各 AI 网页中使用自己的账号

## 5. 界面设计

### 5.1 整体布局

应用窗口从上到下分为三个区域：

- **顶部工具栏（48px）**：应用控制按钮区，包含布局切换、地址栏（单屏模式）、书签、设置
- **内容区（自适应高度）**：WebView 分屏区域，根据选择的布局模式展示 1 / 2 / 3 / 4 个格子
- **底部输入区（80px，分屏模式常驻）**：统一输入框 + 发送按钮 + 格子激活状态指示

### 5.2 顶部工具栏

- 左侧：前进 / 后退 / 刷新按钮
- 中间：地址栏——始终显示当前聚焦格子的 URL；用户点击任意格子即切换聚焦，地址栏随之更新；聚焦格子有蓝色 2px 边框高亮；用户在地址栏输入 URL 或搜索词，导航作用于当前聚焦格子
- 右侧：布局切换图标组（单屏 / 左右 / 上下 / 三分 / 四分，图标直观展示分割方式）、「编辑格子」按钮（打开格子网址配置面板）、书签按钮、设置按钮

### 5.3 分屏内容区

- 每个格子之间用 4px 间隔线分隔，可拖拽调整各格子宽高比例
- 每个格子左上角悬浮显示该格子当前网站的 favicon + 域名，半透明，不遮挡内容
- 用户点击某个格子时，该格子边框高亮（蓝色 2px 描边），表示当前聚焦格子
- 每个格子右上角悬浮菜单（hover 显示）：刷新 / 修改地址 / 在新标签页打开 / 静音

### 5.4 底部统一输入区

- 输入框占据底部区域的大部分宽度，支持多行输入（Shift+Enter 换行，Enter 发送）
- 输入框左侧显示当前激活格子的图标组，用户可点击取消某个格子的勾选（不向该格子发送）
- 右侧发送按钮，发送中显示加载动画，所有格子注入完成后恢复
- 发送后底部输入框自动清空，保留发送历史可通过上箭头键召回

### 5.5 视觉风格

- 基调：简洁、专业，默认跟随系统深色/浅色模式
- 主色调：蓝色 `#1A56DB` 作为强调色
- 字体：系统默认字体（`-apple-system` / `Segoe UI`）
- 格子边框和工具栏保持轻量感，最大化内容展示区域

## 6. 用户核心使用流程

### 6.1 首次使用

1. 下载并安装 `.dmg` 或 `.exe`，打开应用
2. 首次启动弹出欢迎引导：从内置模板中选择一个（如「中美四强」），或自定义每个格子加载的 AI 网站；选定后对应数量的格子自动创建并加载
3. 用户在每个格子里分别登录自己的 AI 账号（与在浏览器里登录体验完全一致）
4. 登录完成后，在底部输入框输入第一个问题，按 Enter，四个 AI 同时开始回答

### 6.2 日常使用

1. 打开应用（各格子自动恢复上次会话或加载预设网址）
2. 根据需求选择布局（单屏普通浏览 / 双屏对比 / 三屏或四屏讨论），各格子加载的网站可随时在「编辑格子」面板中调整
3. 在底部输入框提问，各 AI 同步作答
4. 直接在某个格子里继续追问（点击格子后在该 AI 网页内输入），或继续用底部输入框同步追问
5. 单屏模式下当普通浏览器使用，底部输入框可折叠

## 8. 已知风险与应对

| 风险 | 应对策略 |
|-----|--------|
| AI 官网 UI 更新导致注入失效 | 适配器模块化隔离；应用支持自动更新推送修复；回退到「手动 Enter」保底 |
| 部分 AI 网站检测并拦截自动化操作 | 注入方式模拟真实用户事件（非 Puppeteer），降低被识别概率；必要时在文档中说明手动操作 |
| WebContentsView 内存占用高（4个浏览器实例） | 对不可见格子启用背景节流；提供「暂停」格子的选项 |
| macOS Intel 与 Apple Silicon 兼容性差异 | CI 在两种架构上分别构建测试；发布 Universal Binary |
| Windows 上 WebView2 与 Electron Chromium 冲突 | Electron 自带 Chromium，不依赖系统 WebView2，无冲突风险 |
| Google 账号登录在内嵌格子中被拒绝（Google 全面禁止嵌入式浏览器登录 Google 账号，属于 Google 既定策略，无法绕过） | 检测到拦截页后在格子上方提示，引导用户改用邮箱/手机号登录该 AI 网站；不支持非 Google 登录方式的网站暂列为已知限制 |

## 10. 关于 Chrome 扩展的说明

MultiMind Browser 基于 Electron（内嵌 Chromium），**不支持从 Chrome Web Store 直接安装 Chrome 扩展**。

根本原因：Chrome Web Store 的扩展安装依赖 Google 的签名验证机制，Electron 不是 Google Chrome，无法绕过。市面上真正支持 Chrome Web Store 的浏览器（Arc、Brave、Edge）均直接基于 Chromium 源码构建，维护成本极高，不适合当前阶段。

同时，MultiMind Browser 的核心能力——用 `WebContentsView` 将多个 AI 网站作为独立浏览器实例并排展示——正是绕过了各 AI 网站的 `X-Frame-Options: DENY` 限制才得以实现。这个能力与 Chrome 扩展生态天然不兼容，两者存在架构层面的根本矛盾。

**产品定位因此明确为**：一个专注于多 AI 并行讨论的浏览器工具，而非 Chrome 的替代品。用户在需要 Chrome 扩展时继续使用 Chrome，MultiMind Browser 承担多 AI 深度讨论这一特定场景。

## 11. 未来路线展望

### 第二期：核心体验深化

- **本地记忆系统**：将每次讨论结果通过 SQLite + sqlite-vec 永久存储到本地，支持语义检索，数据完全属于用户
- **综合对比面板**：第三个常驻区域展示多 AI 回答的共识与分歧点，用户可标注值得尊重的回答
- **用户脚本系统**（插件生态第一层）：允许用户编写自定义 JS 脚本注入到指定网站，覆盖划词翻译、页面样式修改、快捷操作等场景，类似 Tampermonkey。基于现有 `executeJavaScript()` 接口扩展，架构改动小

### 第三期：MultiMind 原生插件生态

设计专属插件 API，插件能力包括：

- 在分屏格子里注入 UI 元素
- 监听底部输入框的发送事件，在发送前后做自定义处理
- 读写本地记忆系统
- 调用 MultiMind 专有能力（获取所有格子 URL、跨格子搜索等）

插件格式参考 Chrome 扩展的 `manifest.json` + JS 文件结构，降低开发者学习成本。这些能力是 Chrome 扩展做不到的，因为 Chrome 没有「多个 AI 格子」这个概念，天然形成差异化。

典型插件示例：
- AI 辩论记录插件：自动把四个格子的回答整理成对比表格并存入记忆系统
- 提问增强插件：在底部输入框旁边加「优化我的问题」按钮
- 跨格子搜索插件：在所有格子里同时高亮关键词

### 第四期：AI 伙伴

- **宠物形象 + 语音交互**：浏览器侧边栏入口，语音输入问题，宠物同步复述各 AI 观点并提炼结论
- **自我进化**：分析用户历史交互偏好，自动优化分发策略和提示词

## 12. 插件系统开发规范预留

为后续插件系统留口子，当前开发阶段需遵守以下规范：

- `windowManager.ts` 中的 `executeJavaScript()` 调用必须通过统一的 `injectScript(cellId, script)` 方法封装，不允许在业务代码中直接调用 `webContents.executeJavaScript()`
- 每个 WebContentsView 的生命周期事件（加载完成、URL 变化、标题变化）必须通过 IPC 广播，不允许在主进程内直接处理业务逻辑
- 用户数据存储统一走 `electron-store`，key 命名空间化（如 `browser.*`、`cells.*`），为插件的独立存储命名空间（如 `plugins.pluginId.*`）预留位置
