# MultiMind Flow 格子角色与工作流方案

## 1. 核心判断

MultiMind Flow 的每个格子不只代表一个 AI 网站，也可以代表一个稳定的讨论角色。

第一阶段的价值是“多个 AI 同屏讨论”。第二阶段继续往前走时，真正需要补上的不是更多网站，而是让用户能方便地定义“这个格子在这轮讨论里负责什么”。同一个 ChatGPT 格子可以是反驳者，也可以是总结者；同一个 DeepSeek 格子可以是事实核查者，也可以是技术评审者。角色应绑定在格子会话上，而不是绑定在某个网站或模型上。

可借鉴的方向是“角色化 Agent + 工作流编排 + 过程追踪”，但 MultiMind Flow 不应照搬后端 Agent 平台的实现。当前产品仍以用户自己的 AI 官网账号为基础，通过 WebContentsView 和站点适配器完成注入、读取、转发和总结。

## 2. 产品目标

让用户在一次多 AI 讨论中，可以快速完成这些动作：

- 给每个格子指定讨论角色，例如“主答者”“反驳者”“事实核查者”“技术评审者”“总结者”。
- 转发时自动根据目标格子的角色生成合适任务指令，不需要用户反复手写“请你评价/反驳/补充”。
- 使用预设工作流把常见讨论路径一键串起来，例如“多方首答 → 互评 → 总结成文档”。
- 保留清晰的讨论溯源，让最终文档知道哪些格子、哪些角色、哪些转发链路参与了形成结论。

## 3. 角色模型

### 3.1 Role Template

新增“角色模板”概念，每个模板定义这个格子在讨论中的任务倾向。

建议首批内置角色：

| 角色 | 用途 | 典型任务指令 |
|---|---|---|
| 主答者 | 给出完整初稿或方案 | 直接回答问题，给出结构化方案和关键假设 |
| 反驳者 | 找漏洞、挑战结论 | 找出遗漏、错误、过度假设和反例 |
| 事实核查者 | 检查事实、时间、来源 | 核查具体事实，标出不确定信息和需要验证的地方 |
| 技术评审者 | 评审实现方案 | 从架构、复杂度、可维护性、风险和测试角度审查 |
| 产品评审者 | 评审用户价值和交互 | 从目标用户、场景、体验和优先级角度审查 |
| 总结者 | 收敛讨论成果 | 综合各方观点，输出清晰 Markdown 文档 |
| 执行规划者 | 面向第三阶段执行 | 把结论拆成可执行任务、命令、文件变更和验收标准 |

角色模板不替代用户 prompt。它只提供“格子职责”和“转发/总结时的任务框架”，最终 prompt 仍由用户输入和当前讨论上下文组成。

### 3.2 自定义角色

用户可以新增自定义角色，字段建议保持轻量：

```typescript
interface CellRoleTemplate {
  id: string;
  name: string;
  description: string;
  promptInstruction: string;
  outputPreference?: "bullets" | "markdown" | "review" | "plan";
  builtIn: boolean;
}
```

自定义角色本地保存即可。短期可放入 `electron-store`，长期记忆系统接入后再迁移到 SQLite。

### 3.3 格子角色状态

角色是格子的“当前讨论状态”，不是网站配置。

```typescript
interface CellRoleState {
  cellId: string;
  roleId: string | null;
  customInstruction?: string;
}
```

默认可以为空。为空时保持现有行为：统一发送和转发仍按通用 prompt 处理。

## 4. 交互方案

### 4.1 格子标题栏入口

每个格子标题栏增加一个“角色”入口，显示当前角色名。点击后打开轻量菜单：

- 无角色
- 主答者
- 反驳者
- 事实核查者
- 技术评审者
- 产品评审者
- 总结者
- 自定义...

在四分屏中，角色名应短，避免挤占标题栏。完整说明放在 tooltip 或配置面板中。

### 4.2 模板创建时指定角色

现有布局模板只定义布局和站点。后续可扩展为“讨论模板”：

```typescript
interface DiscussionTemplate {
  id: string;
  name: string;
  layout: LayoutMode;
  panes: Array<{
    cellId: string;
    siteId: string;
    roleId?: string;
  }>;
}
```

示例：

| 模板 | 格子配置 |
|---|---|
| 四方交叉验证 | Claude=主答者，ChatGPT=反驳者，DeepSeek=技术评审者，豆包=总结者 |
| 产品方案评审 | Claude=产品评审者，ChatGPT=技术评审者，DeepSeek=反驳者，豆包=总结者 |
| 事实核查 | Perplexity=事实核查者，ChatGPT=主答者，Claude=反驳者 |

布局模板和讨论模板可以并存。布局模板解决“打开哪些网站”，讨论模板解决“这些格子分别扮演什么角色”。

### 4.3 底部统一输入

统一发送时，可以有两种模式：

- 普通模式：保持当前行为，把用户输入原样发送到所有活跃聊天格子。
- 角色模式：给每个活跃格子追加该格子的角色指令。

建议默认仍使用普通模式，避免用户无感知地改变 prompt。角色模式通过底部输入旁的开关启用，文案可为“按角色发送”。

### 4.4 转发行为

转发是角色系统最有价值的入口。

当前转发 prompt 主要表达“请评价最后一条 AI 回答”。引入角色后，应根据目标格子的角色调整任务说明：

- 目标是反驳者：请重点找漏洞、反例、遗漏和不成立的假设。
- 目标是事实核查者：请只核查事实性陈述，区分已知事实、推断和不确定项。
- 目标是技术评审者：请从架构、实现复杂度、风险、测试和维护成本审查。
- 目标是总结者：请综合上下文，沉淀为结构化结论，不展开新话题。

重要边界：

- 发给 AI 的 prompt 只包含任务所需上下文和角色任务说明。
- 给用户看的不确定性提示仍走 `SHOW_CELL_NOTICE`，不能混进 prompt。
- 裁切提示仍同时进入 prompt 和界面 notice。
- 多跳转发时，角色任务说明只包裹最外层一次，不能嵌套进时间线。

## 5. 工作流方案

### 5.1 Workflow Preset

在角色基础上引入“工作流预设”。工作流不是后台自动 Agent 系统，而是帮用户批量触发现有的发送、等待、转发、总结动作。

```typescript
interface DiscussionWorkflowTemplate {
  id: string;
  name: string;
  steps: DiscussionWorkflowStep[];
}

type DiscussionWorkflowStep =
  | { type: "send-to-roles"; roleIds: string[] }
  | { type: "forward"; fromRoleId: string; toRoleId: string }
  | { type: "forward-all-to"; targetRoleId: string }
  | { type: "generate-document"; summarizerRoleId: string };
```

首批只做半自动：执行每一步前展示确认，用户可跳过。这样能保持用户对讨论质量的控制，也避免全自动交叉验证制造大量低价值内容。

### 5.2 推荐工作流

**四方交叉验证**

1. 用户输入问题，发送给主答者、反驳者、技术评审者。
2. 用户选择最有价值的回答，转发给反驳者或技术评审者。
3. 用户把关键上下文转发给总结者。
4. 总结者输出 Markdown 文档。

**产品方案评审**

1. 主答者生成方案。
2. 产品评审者检查目标用户、场景和价值。
3. 技术评审者检查实现边界、风险和测试。
4. 反驳者挑战关键假设。
5. 总结者输出最终产品方案。

**技术执行计划**

1. 主答者生成技术方案。
2. 技术评审者审查架构。
3. 执行规划者拆成任务清单、命令、文件变更和验收标准。
4. 总结者沉淀为可交给第三阶段终端/Agent 使用的 Markdown。

## 6. 溯源与记忆

角色系统需要和时间线结合，而不是只做 UI 标记。

建议扩展时间线条目：

```typescript
interface TimelineEntry {
  role: "user" | "assistant";
  content: string;
  source: "bottom-input" | "forward-injection" | "document-generation" | "dom-detected";
  timestamp: number;
  cellRoleId?: string;
  workflowRunId?: string;
  forwardRecordId?: string;
}
```

这样后续总结文档可以包含更可靠的元信息：

- 参与的 AI 网站
- 每个格子的角色
- 哪些转发链路被用户触发
- 哪些内容发生过裁切
- 最终文档由哪个角色生成

长期记忆接入后，不建议保存所有中间聊天全文为长期知识。更合理的做法是：

- 保存最终 Markdown 文档。
- 保存轻量 trace metadata，方便回溯文档如何形成。
- 中间过程只作为本次会话上下文和调试溯源，不默认进入长期知识库。

## 7. 技术落地顺序

### 阶段 A：角色模板与格子角色

- 新增 `presetRoles.ts`，定义内置角色模板。
- 在共享类型中加入 `CellRoleTemplate`、`CellRoleState`。
- 在 `BrowserState` 中加入 `cellRoles`。
- 用 `electron-store` 持久化角色状态和自定义角色。
- 在格子标题栏加入角色选择入口。

这一阶段不改变发送逻辑，只让用户能定义角色。

### 阶段 B：角色化转发

- `buildForwardPrompt` 接收目标格子的 `roleId`。
- 按目标角色选择任务说明文案。
- 维持现有语言检测逻辑：prompt 语言跟随对话内容，不跟随界面语言。
- 转发记录中加入源/目标角色信息。

这一阶段能直接提升交叉验证体验，收益最大。

### 阶段 C：角色化统一发送

- 底部输入增加“按角色发送”开关。
- 开启后，活跃格子收到“用户问题 + 本格子角色任务说明”。
- 关闭时保持现有原样发送行为。

这一阶段需要谨慎，因为它会改变首轮回答风格。

### 阶段 D：讨论模板

- 扩展模板系统，支持站点 + 角色组合。
- 保留现有布局模板，不破坏当前用户习惯。
- 模板应用后同时设置布局、站点和角色。

### 阶段 E：半自动工作流

- 新增工作流模板。
- 每一步由用户确认执行。
- 执行过程写入 `workflowRunId` 和 trace metadata。
- 后续再考虑自动连续执行，但不作为首版目标。

## 8. 不做的事

- 不把角色绑定到具体 AI 网站。角色属于格子会话，网站只是承载者。
- 不引入 Express、Prisma、PostgreSQL 或云端后端。
- 不把所有中间讨论默认写入长期记忆。
- 不做全自动 N 对 N 交叉验证作为默认流程。
- 不在 `WindowManager` 里写具体角色分支。角色 prompt 组装应封装为独立模块。
- 不改变现有站点适配器边界。适配器仍只负责读写某个网站，不理解产品角色。

## 9. 推荐下一步

优先做“阶段 A + 阶段 B”。

这两步能用较小改动把“多个格子角色扮演”落到真实工作流里：用户先给格子定义角色，然后转发时目标格子自动按角色评价。它不要求新增数据库、不要求改动站点适配器、不影响现有统一发送稳定性，和当前第二阶段的时间线、转发、总结能力衔接最自然。
