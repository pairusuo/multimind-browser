# MultiMind
## 从多 AI 讨论到产品落地的一体化工作台 — 产品设计文档 v0.2
**2026年6月 · 机密 · 内部使用**

> 本版本相较 v0.1 的核心变化：产品名称由 MultiMind Browser 改为 **MultiMind**，
> 因为浏览器只是产品的第一阶段形态，不是最终定位；新增第二、三阶段的长期规划；
> 原 v0.1 中已完成的 MVP 内容（多 AI 分屏浏览器）整理为「第一阶段」章节保留。

---

## 1. 产品愿景

### 1.1 一句话定位

MultiMind 是一个让用户"先和多个 AI 讨论清楚，再把讨论结果变成真实产品"的一体化工作台。

### 1.2 为什么不是"做一个浏览器"

如果 MultiMind 只是一个能同时打开多个 AI 网页的分屏浏览器，护城河很浅——这件事本身没有不可替代的技术壁垒，竞争力有限。真正的产品价值在于把三个原本割裂的环节串成一条线：

1. **讨论**：和多个 AI 交叉验证一个想法，看到不同模型的分歧和共识
2. **沉淀**：把讨论收敛成一份结构化文档，而不是讨论完就忘
3. **执行**：用 Agent 能力把文档变成真实代码或完成真实任务

市面上没有产品把这三层完整打通。Claude Code 和 Codex 只做第3层，且绑定单一模型；普通浏览器只能做第1层的一半（开多个标签页，但不能同步提问、不能沉淀）。

### 1.3 三阶段产品形态

| 阶段 | 形态 | 核心能力 | 状态 |
|-----|------|---------|------|
| 第一阶段 | 多 AI 分屏浏览器 | 同时打开多个 AI/搜索引擎官网，统一输入同步发送，用户用自己的账号登录讨论 | **MVP 已完成** |
| 第二阶段 | 讨论 → 文档沉淀 | AI 互相查看彼此回答，交叉验证，用户指定一个 AI 汇总成结构化文档，长期记忆持久化存储 | 规划中 |
| 第三阶段 | 浏览器 + 终端 + Agent | 内置真实终端（可执行 Codex CLI 等用户已安装的工具），内置 Agent 能力，用文档驱动写代码（类似 Claude Code/Codex），或处理通用任务（类似 OpenClaw/Hermes），宠物形象作为情绪价值层 | 远期规划 |

三个阶段不是互相替代，而是逐步叠加——第一阶段的浏览器形态会一直保留并持续优化，第二、三阶段是在此基础上新增的能力层。

## 2. 第一阶段：多 AI 分屏浏览器（MVP，已完成）

### 2.1 产品定位

MultiMind 第一阶段形态是一款基于 Electron（内嵌 Chromium）的桌面浏览器，核心能力是将浏览器窗口分割为多个独立的 WebView 格子，让用户同时打开多个 AI 网页（Claude、ChatGPT、DeepSeek 等）或搜索引擎，并通过底部统一输入框一次输入、同步发送到所有窗口。

本工具同时也是一个完整可用的浏览器，用户可以像使用 Chrome/Safari 一样用它访问任意网站。

### 2.2 产品背景与真实痛点

Claude Code 与 Codex 桌面版均仅支持 Apple Silicon（M 系列芯片），2020 年前购买的所有 Intel 芯片 Mac 用户均无法使用其桌面应用。同时，这两款工具均为单一模型绑定：Claude Code 只用 Claude，Codex 只用 GPT。它们的记忆是项目级的，不是个人级的。

MultiMind 从三个维度切入：支持 Intel Mac；同时展示多个 AI 的视角；属于用户自己的浏览器工具，而不是某家 AI 厂商的锁定商品。

### 2.3 目标用户

- 需要同时参考多个 AI 意见来深入思考问题的知识工作者
- 因硬件限制（Intel Mac）无法使用 Claude Code / Codex 桌面版的用户
- 希望在多个 AI 之间进行横向对比而无需反复切换标签页的用户
- Windows 用户（同样没有官方多 AI 并行工具）

### 2.4 已实现功能清单

**分屏布局**：单屏 / 左右分屏 / 上下分屏 / 三分屏 / 四分屏五种模式，格子间可拖拽调整比例，快捷键切换。

**统一输入与同步发送**：底部常驻输入框，一次输入同步注入所有激活格子并触发发送；各 AI 网站独立的注入适配器（详见 adapter-reference.md）；注入失败时有统一的提示系统兜底。

**用户自定义格子**：每个格子加载哪个网站完全由用户决定，内置 AI 网站清单（Claude、ChatGPT、DeepSeek、Kimi、文心一言、通义千问、豆包、智谱清言等）和搜索引擎清单（Google、Bing、百度、搜狗、DuckDuckGo），支持自定义 URL；系统自动判定网站类型（Chat 模式或 Search 模式），不需要用户手动标注。

**通用浏览器能力**：标准地址栏、前进后退刷新、多标签页管理、书签、下载管理、开发者工具、隐私模式。

**已知限制**：Gemini 因仅支持 Google 账号登录而无法使用（Google 全面禁止嵌入式浏览器登录，无法绕过）；不支持 Chrome Web Store 扩展（Electron 与 Chrome Web Store 的签名机制不兼容，详见技术方案中的说明）。

**跨平台**：macOS Universal（同时支持 Intel 和 Apple Silicon）+ Windows x64，当前版本不做代码签名（MVP 阶段决策，用户需手动绕过系统安全提示）。

> 第一阶段的详细技术实现规范（适配器开发、注入机制、IPC 设计等）由 `AGENTS.md` 维护，本文档不重复列出实现细节，只保留产品层面的功能描述。

## 3. 第二阶段：讨论 → 文档沉淀

### 3.1 核心场景

用户带着一个想法或问题，在 MultiMind 里完成以下流程：

1. 在多个格子里同时向不同 AI 提问（第一阶段已具备的能力）
2. 各 AI 给出各自的初步回答
3. **交叉验证**：MultiMind 自动把 A 的回答转述给 B、C、D，请它们评论、补充或反驳；同理对每个 AI 都做一轮交叉
4. 用户从交叉验证后的多方观点中，指定一个 AI 作为"总结者"
5. 总结者 AI 把"原始问题 + 第一轮各自回答 + 第二轮交叉意见"整合成一份结构化文档
6. 这份**最终文档**被持久化存储为长期记忆，第一轮和第二轮的中间过程不单独保留

### 3.2 为什么只存最终结果，不存过程

讨论过程中的中间回答价值有限且占用空间，真正有长期价值的是收敛后的结论。这个设计让长期记忆保持精炼，避免随着使用时间增长，记忆库被大量低价值的中间内容淹没。

### 3.3 技术路径：交叉验证如何实现

这一步是第二阶段最关键也最有技术挑战的环节。第一阶段的注入适配器只解决了"写入"（把文字填进输入框、触发发送），第二阶段需要新增"读取"能力：

1. **读取适配器**：为每个 AI 网站新增一套逻辑，能够提取该网站当前对话中最新一条 AI 回答的文本内容，并判断该回答是否已经生成完毕（不是流式输出中途的不完整内容）
2. **交叉注入**：把 AI A 的回答文本，转述注入到 AI B 的输入框中（措辞类似"这是另一个 AI 的观点：{内容}，你怎么看，有没有需要补充或反驳的地方"），触发新一轮发送
3. **状态编排**：整个交叉验证流程涉及多个格子的"等待生成完毕"状态判断，是技术上最脆弱的环节，需要针对每个网站单独处理生成完成的检测逻辑，并设置合理的超时与异常回退

技术细节由 `adapter-reference.md` 中的「读取适配器」章节维护。

### 3.4 长期记忆系统

**存储原则**：本地优先，所有记忆数据存储在用户设备本地，不上传任何服务器，保证隐私和数据主权。

**存储内容**：每次讨论沉淀的最终文档，包含原始问题、参与讨论的 AI 列表、最终文档正文、生成时间、用户指定的标签（可选）。

**存储形式**：SQLite 数据库 + 全文检索（FTS5），文档内容支持后续语义检索，方便用户在新的讨论中回溯过去沉淀的经验和知识。

**记忆的用途**：
- 用户随时可以在本地知识库中检索过去的讨论文档
- 未来开始新的相关讨论时，可以选择性地把过去的某份文档作为背景信息提供给 AI
- 第三阶段的 Agent 执行时，可以读取相关的历史文档作为上下文参考

### 3.5 触发方式

交叉验证和文档生成不是自动触发的，需要用户主动操作（避免不必要的 API/网站调用消耗）：
- 各格子完成首轮回答后，界面上出现"开始交叉讨论"的操作入口
- 交叉验证完成后，界面提供"生成总结文档"入口，用户选择由哪个 AI 担任总结者
- 文档生成后，用户确认无误后手动保存进长期记忆（也可以选择不保存）

## 4. 第三阶段：浏览器 + 终端 + Agent

### 4.1 核心场景

第二阶段产出的文档，不应该止步于"存起来"，而是可以驱动实际行动：

- **写代码场景**：用户拿着第二阶段沉淀的设计文档，驱动一个内置 Agent（类似 Claude Code / Codex 的工作方式）读取文档、生成代码、操作指定的本地文件目录，把想法变成可运行的产品
- **通用任务场景**：类似 OpenClaw / Hermes 的定位，Agent 处理更广泛的任务（不仅是写代码），具备工具调用能力（文件读写、终端命令、网页访问等）
- **直接使用已安装 CLI 工具场景**：用户不一定需要 MultiMind 自己重新实现一套写代码能力，也可以在 MultiMind 内置的真实终端里直接运行自己电脑上已经装好的 Codex CLI、Claude Code CLI 等工具，效果等同于打开了一个独立终端窗口

### 4.2 与第一、二阶段的关系

第一、二阶段（浏览器分屏讨论、文档沉淀）依然保留并独立可用——不是所有用户都需要 Agent 执行能力，很多用户可能只需要"和多个 AI 讨论 + 沉淀知识"这两层。第三阶段是在此基础上叠加的高级能力，面向有明确执行需求（写代码、做任务）的用户。

### 4.3 关键技术差异：两套完全不同的实现机制

这是第三阶段最重要的架构判断，需要清楚区分：

| | 第一、二阶段（讨论层） | 第三阶段（Agent 执行层） |
|---|---|---|
| 技术实现 | WebContentsView 注入官网页面，模拟用户操作 | 真正的 API 调用（类似 Claude Code 的实现方式），或代理调用用户已安装的 CLI 工具 |
| 账号体系 | 用户登录自己的官网账号（免费版或自己付费订阅的版本） | 用户提供自己的 API Key，或使用自己已登录的 CLI 工具账号 |
| 适用场景 | 探索性讨论、交叉验证、低频次深度思考 | 高频、需要工具调用、需要精确控制上下文的执行任务 |
| 成本承担方 | 用户自己的官网订阅（如有） | 用户自己的 API 用量费用，或自己的 CLI 工具订阅 |

两套机制在同一个应用里共存，通过「长期记忆」这一层打通——讨论层沉淀的文档，可以作为 Agent 执行层的输入上下文。

### 4.4 内置终端：一体化工作台的基础设施

**核心定位**：在 MultiMind 窗口内嵌入一个真实可用的终端（效果等同于 iTerm/Terminal 嵌在应用里），用户可以在里面直接运行任何命令行工具，包括 `codex`、`git`、`npm` 等，不受任何限制和包装。

**为什么先做这个，而不是先做"MultiMind 自己的 Agent 写代码能力"**：内置终端是一个相对独立、工程边界清晰的能力，一旦做出来，"调用用户已安装的 Codex CLI 进行代码开发"这件事基本就顺带实现了——不需要重新发明一套命令执行框架。而"MultiMind 自己直接调 API 实现类似 Claude Code 的能力"，是更重的工程投入，优先级排在内置终端之后。

**与分屏格子体系的关系**：终端不是浏览器格子的一种类型，底层完全独立管理（终端管理的是 PTY 进程的生命周期，格子管理的是 WebContentsView 的生命周期，两者无共享逻辑）。但在视觉呈现上，终端可以出现在分屏网格的任意一个位置——例如四分屏布局下，三个格子用于多 AI 讨论，第四个位置切换为终端，让"讨论"和"执行"出现在同一屏画面里。

**当前阶段的边界（有意收窄，避免一次性做太多）**：
- 只做终端本身的接入，不做"讨论结果自动传递给终端"这类数据打通，用户需要的话先靠手动复制粘贴
- 不做代码编辑器（语法高亮、文件树等 IDE 功能），只做终端
- 终端默认是收起还是常驻、能否同时开多个终端标签，留待开发时根据实际体验调整

**技术选型**：`node-pty`（通过 `@homebridge/node-pty-prebuilt-multiarch` 这个提供多平台预编译二进制的分支，避免在三个目标平台上分别处理原生模块编译）+ `xterm.js`（终端渲染前端）。这是当前技术栈里第一个需要原生编译模块的依赖，打包流程需要额外处理（具体见 `AGENTS.md`），第一次接入时应预留排查时间，不假设一次到位。

### 4.5 Agent 技术方案（高层级，待第二阶段验证后细化）

- **API 兼容性**：优先支持 Anthropic、OpenAI、DeepSeek 等主流 API 标准（多数兼容 OpenAI 的请求格式），用户在设置中填入自己的 API Key
- **工具调用能力**：参考 MCP（Model Context Protocol）标准，给 Agent 挂载文件系统、终端、Git 等工具；其中"终端"这个工具可以直接复用 4.4 节的内置终端基础设施，Agent 把命令丢进同一个终端执行，而不是另起一套独立的命令执行机制
- **安全边界**：文件操作必须限定在用户明确授权的目录范围内，每次实际修改前需要用户确认（不做静默写入）
- **长期记忆接入**：Agent 执行任务时可以检索并引用第二阶段沉淀的相关文档，避免每次都从零开始

### 4.6 宠物形象：情绪价值层，非核心功能

宠物形态作为 MultiMind 的一个可选的拟人化交互层，承担的是情绪价值，不是产品的核心竞争力，开发优先级低于上述三层核心能力。可能的形式包括：在 Agent 执行长任务时提供进度反馈的拟人化界面、长期记忆的拟人化"记忆守护者"形象等，具体形态在前三层验证完成后再设计。

## 5. 第一阶段技术方案（详细，沿用 v0.1 内容）

### 5.1 技术选型

| 层次 | 技术选型 / 理由 |
|-----|--------------|
| 应用框架 | Electron 33+  —  内嵌 Chromium，跨平台（Mac x86/ARM、Windows），成熟生态 |
| 前端 UI | React 18 + TypeScript  —  AI 辅助编码友好，组件化易维护 |
| 样式 | Tailwind CSS  —  快速布局，无需手写大量 CSS |
| WebView 技术 | Electron WebContentsView  —  官方推荐替代已废弃的 BrowserView |
| 主进程通信 | Electron IPC（ipcMain / ipcRenderer）  —  渲染进程与主进程双向通信 |
| 网站注入 | webContents.executeJavaScript()  —  向 WebView 注入输入和发送指令 |
| 数据持久化 | electron-store（配置）+ SQLite（长期记忆，第二阶段引入） |
| 打包工具 | electron-builder  —  一键生成 .dmg（x86 + ARM）和 .exe 安装包 |

### 5.2 关于 Chrome 扩展的说明

MultiMind 基于 Electron（内嵌 Chromium），不支持从 Chrome Web Store 直接安装 Chrome 扩展。根本原因是 Chrome Web Store 的扩展安装依赖 Google 的签名验证机制，Electron 不是 Google Chrome，无法绕过；同时 MultiMind 的核心能力（WebContentsView 并排展示多个 AI 网站）依赖绕过各网站的 iframe 限制，这与 Chrome 扩展生态存在架构层面的根本矛盾。产品定位明确为多 AI 并行讨论工具，而非 Chrome 替代品。

### 5.3 隐私与安全

- 第一、二阶段所有用户数据（配置、书签、格子布局、长期记忆文档）存储在本地，不上传任何服务器
- 各格子 WebView 的 Cookie 和 Session 由 Electron 管理，存储在本地用户目录，每个格子独立 partition
- 注入脚本的时序经过自然化处理（随机延迟），目的是让自动化操作更接近真实用户习惯，不涉及绕过任何网站的验证码或双重验证机制
- 第三阶段 Agent 执行层用户自备 API Key，密钥本地加密存储，不经过任何中间服务器转发

### 5.4 跨平台打包

| 平台 | 架构 | 产物格式 |
|-----|-----|--------|
| macOS | x86_64 (Intel) + arm64 (Apple Silicon) | .dmg（Universal） |
| Windows | x64 | .exe（NSIS 安装包） |

当前版本不做代码签名和 Notarization（MVP 阶段决策），用户安装时需手动绕过系统安全提示（Gatekeeper / SmartScreen），详见安装说明文档。

> 完整的实现规范（IPC 设计、组件结构、构建配置等）由 `AGENTS.md` 维护。
